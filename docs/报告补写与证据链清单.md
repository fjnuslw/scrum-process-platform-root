# 报告补写与证据链清单

用于在正式进入 Trae IDE 前，明确哪些材料已经具备，哪些材料必须继续补齐。

## 一、已经具备

- 可运行前后端代码框架
- 核心接口实现
- Vue 前端基础版页面
- 后端自动化测试
- README 运行说明
- AI 错误修正记录
- Git 阶段提交
- 初始化到构建通过的截图
- 报告底稿已补充“当前阶段整理”和“基于 docs 的进一步完善”

## 二、必须继续补齐

- Trae Builder 会话记录 3~5 组
- Trae Builder 模型选择截图
- Trae Builder 的真实代码迭代截图
- 双人配合记录 2~3 组
- Gitee 远程仓库截图
- 录屏脚本
- 最终录屏
- 最终报告中的测试截图与缺陷闭环说明

## 三、推荐补齐顺序

1. 在 Trae Builder 中完成“故事加入迭代改为拖拽式”
2. 在 Trae Builder 中尝试接入 MySQL
3. 在 Trae Builder 中完成 1~2 个 bug 修复闭环
4. 补测试截图 / 补 Postman 材料
5. 推送到 Gitee 并截图
6. 整理双人讨论记录
7. 录屏
8. 回填报告

## 四、报告中最需要替换的占位内容

- 线下过程中的产品代办表、发布计划、验收测试表
- 线上过程中的产品列表、迭代计划、任务看板截图
- 代码仓库部分的 Gitee 首页、分支、提交记录截图
- 双人配合记录
- AI 关键交互示例
- 测试报告截图

## 五、当前最核心的风险

当前最大的风险不是代码功能，而是：

- Trae 证据链不足
- Gitee 远程仓库缺失
- 双人配合材料不足
- MySQL 运行层尚未真正接入

因此接下来不宜再继续无边界扩功能，而应围绕这些风险补证据链。
