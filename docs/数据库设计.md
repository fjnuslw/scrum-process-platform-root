# 数据库设计

## 1. 设计目标

本项目围绕 Scrum 过程管理平台基本功能设计数据库，要求结构简单、关系清晰、便于接口开发与测试。

## 2. 核心表

### 2.1 user_stories

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | 主键 |
| title | varchar(255) | 故事标题 |
| description | text | 用户故事描述 |
| story_points | decimal(4,1) | 故事点数 |
| priority_order | int | 优先级顺序 |
| status | varchar(32) | backlog / todo / in_progress / done |
| sprint_id | bigint null | 所属迭代 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 2.2 sprints

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | 主键 |
| name | varchar(255) | 迭代名称 |
| start_date | date | 开始日期 |
| end_date | date | 结束日期 |
| goal | varchar(500) null | 迭代目标 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 2.3 retrospectives

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | 主键 |
| sprint_id | bigint | 关联迭代 |
| good_1 | varchar(500) | 做得好的第一项 |
| good_2 | varchar(500) | 做得好的第二项 |
| good_3 | varchar(500) | 做得好的第三项 |
| improve_1 | varchar(500) | 待改进的第一项 |
| improve_2 | varchar(500) | 待改进的第二项 |
| improve_3 | varchar(500) | 待改进的第三项 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 2.4 burndown_logs

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | 主键 |
| sprint_id | bigint | 关联迭代 |
| log_date | date | 记录日期 |
| remaining_points | decimal(6,1) | 剩余故事点 |
| created_at | datetime | 创建时间 |

## 3. 约束建议

- `story_points` 只能取课程要求的固定序列：0.5、1、2、3、5、8、20、40
- `priority_order` 必须支持拖拽后批量更新
- `status` 建议限制为：`todo`、`in_progress`、`done`
- 一个用户故事在任意时刻最多属于一个 sprint
- 一个 sprint 最多对应一条 retrospective 记录，或允许覆盖更新

## 4. 设计说明

- 不单独拆 backlog 表，直接通过 `user_stories.priority_order` 表示顺序
- 不单独拆 story history 表，先用 `burndown_logs` 记录燃尽图所需的每日剩余点
- 当前不实现复杂权限、成员和任务拆分，保持课程作业规模可控
