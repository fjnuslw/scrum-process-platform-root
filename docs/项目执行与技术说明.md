# Scrum过程管理平台项目执行与技术说明

## 1. 文档目的

本文档用于指导个人主导完成《AI辅助软件过程实验》中的 Web 版 Scrum 过程管理平台开发，兼顾以下目标：

1. 明确项目范围，避免过度开发。
2. 作为本地开发、Codex 检查、Trae IDE 辅助生成代码时的统一上下文文档。
3. 帮助整理开发顺序、交付材料、测试范围和报告写作要点。
4. 在 Builder 模式下，为个人独立推进项目提供可执行的实施方案。

> 建议将本文件放在项目根目录，并在使用 Codex / Trae IDE 时先让 AI 阅读本文件后再执行具体任务。

---

## 2. 先做什么

### 2.1 模式选择

本项目建议选择 **Builder 模式**。

原因：
- Builder 模式允许两人前期讨论、后续各自独立开发。
- 代码仓库、AI 会话记录、Git 提交记录均可独立保存。
- 相比 Solo + OpenSpec，Builder 模式对队友的持续配合依赖更低，更适合个人主导推进。

### 2.2 开发目标

本项目优先完成 **基本版功能**，不主动开发增强版，除非基本版、测试、录屏、报告都已完成。

### 2.3 开发原则

1. 先完成核心功能，再优化 UI。
2. 先保证能运行，再考虑美观和扩展。
3. 每完成一个小功能就提交 Git。
4. 每个关键功能都保留 AI 辅助记录。
5. 主动记录 AI 生成错误案例，至少 2 个。

## 3. 技术文档是什么

### 3.1 在本项目里，“技术文档”可以理解为

用于说明项目如何设计、如何实现、如何测试的结构化说明文件。它不是空泛介绍，而是给自己、给 AI、给老师看的“项目说明书”。

### 3.2 这类技术文档通常包含

- 项目目标
- 功能范围
- 技术栈说明
- 页面与模块划分
- 数据模型设计
- 接口设计
- 测试范围
- 开发约束
- 交付要求

### 3.3 为什么要写

1. **给自己看**：避免开发过程中跑偏。
2. **给 AI 看**：让 Codex / Trae IDE 理解你的项目边界、命名规范、实现顺序。
3. **给报告用**：后续写实验总结时可以直接引用，不必临时回忆。

### 3.4 本文档的作用

本文档本身就可以作为你的第一版技术文档使用。

## 4. 推荐技术栈

为降低个人开发风险，建议采用以下技术栈：

### 前端
- Vue 3
- Vite
- Element Plus
- Vue Router
- Pinia
- ECharts（用于燃尽图）
- `@dnd-kit` 或 `sortablejs` 的 Vue 方案（用于拖拽）

### 后端
- Node.js
- Express
- Prisma 或 Sequelize（二选一，推荐 Prisma）

### 数据库
- MySQL

### 测试
- Postman + Newman
  或
- Jest + Supertest

### 版本管理
- Git
- Gitee 远程仓库

### 选择理由
- 技术资料多，适合课程项目。
- CRUD 开发速度快。
- UI 组件丰富，减少手写样式成本。
- 前后端职责清晰，便于 AI 辅助生成。
- 接口测试工具成熟，便于快速完成自动化测试。

## 5. 项目范围

## 5.1 必做功能（基本版）

### 1）用户故事管理
支持：
- 新增用户故事
- 编辑用户故事
- 删除用户故事
- 展示字段：
  - 标题
  - 描述（作为…我想要…以便…）
  - 故事点数（0.5、1、2、3、5、8、20、40）

### 2）产品待办列表管理
支持：
- 按优先级排序展示用户故事
- 显示标题与故事点数
- 拖拽调整优先级并保存
- 筛选：全部 / 待开发

### 3）迭代计划管理
支持：
- 创建迭代（名称、起止日期）
- 将产品待办中的用户故事加入当前迭代

### 4）迭代开发管理
支持：
- 看板三列：未开始 / 进行中 / 已完成
- 拖拽切换状态
- 生成燃尽图：
  - 计划线：按总故事点数和迭代天数线性生成
  - 实际线：根据故事完成状态变化推算剩余点数

### 5）迭代回顾管理
支持：
- 记录做得好的三件事
- 记录待改进的三件事
- 查看历史迭代回顾

## 5.2 不做内容

为保证按时交付，本项目默认不做以下增强功能：
- 故事拆分 / 合并
- Epic 层级管理
- 技术任务分解
- 问题追踪系统
- 用户故事地图
- 缺陷与重构专题模块

## 6. 页面规划

建议页面结构如下：

1. `/stories` 用户故事管理页
2. `/backlog` 产品待办列表页
3. `/sprints` 迭代管理页
4. `/board/:sprintId` 迭代看板页
5. `/burndown/:sprintId` 燃尽图页
6. `/retrospectives` 回顾列表页
7. `/retrospectives/:sprintId` 某次迭代回顾页

可做成左侧菜单 + 右侧内容区布局。

## 7. 数据模型设计

## 7.1 UserStory

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int | 主键 |
| title | varchar | 标题 |
| description | text | 用户故事描述 |
| storyPoints | decimal | 故事点 |
| priority | int | 优先级，数值越小越靠前 |
| status | enum | TODO / IN_PROGRESS / DONE |
| sprintId | int/null | 所属迭代 |
| createdAt | datetime | 创建时间 |
| updatedAt | datetime | 更新时间 |

## 7.2 Sprint

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int | 主键 |
| name | varchar | 迭代名称 |
| startDate | date | 开始日期 |
| endDate | date | 结束日期 |
| createdAt | datetime | 创建时间 |
| updatedAt | datetime | 更新时间 |

## 7.3 Retrospective

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int | 主键 |
| sprintId | int | 对应迭代 |
| good1 | text | 做得好的第1项 |
| good2 | text | 做得好的第2项 |
| good3 | text | 做得好的第3项 |
| improve1 | text | 待改进第1项 |
| improve2 | text | 待改进第2项 |
| improve3 | text | 待改进第3项 |
| createdAt | datetime | 创建时间 |
| updatedAt | datetime | 更新时间 |

## 7.4 BurndownSnapshot（推荐）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int | 主键 |
| sprintId | int | 对应迭代 |
| date | date | 日期 |
| remainingPoints | decimal | 当日剩余故事点 |
| createdAt | datetime | 创建时间 |

> 说明：如果不想做自动快照，也可以在故事状态变化时重新计算当前剩余点数并生成前端图表数据，但保留快照表更方便测试与展示。

- `POST /api/sprints`
- `GET /api/sprints/:id`
- `PUT /api/sprints/:id`
- `DELETE /api/sprints/:id`

## 8.3 回顾接口

- `GET /api/retrospectives`
- `GET /api/retrospectives/:sprintId`
- `POST /api/retrospectives`
- `PUT /api/retrospectives/:id`

## 8.4 燃尽图接口

- `GET /api/sprints/:id/burndown`

返回建议：
```json
{
  "sprintId": 1,
  "planned": [
    { "date": "2026-05-01", "remaining": 13 },
    { "date": "2026-05-02", "remaining": 11.7 }
  ],
  "actual": [
    { "date": "2026-05-01", "remaining": 13 },
    { "date": "2026-05-02", "remaining": 13 }
  ]
}
```

### 推荐提交示例

## 11. AI 辅助开发记录要求

至少记录以下内容：

1. 需求拆解类对话
2. 页面或接口生成类对话
3. Bug 修复类对话
4. 测试脚本生成类对话
5. 至少 2 个 AI 错误案例

### 每个案例建议写法
- 指令内容
- AI 输出结果
- 问题现象
- 人工发现方式
- 修正方法
- 修正后效果

### 适合记录的典型错误
- 字段名不一致导致接口报错
- 删除后列表未刷新
- 拖拽排序未真正持久化
- 状态更新成功但燃尽图未同步
- 测试脚本断言错误

## 12. 接口自动化测试建议

## 12.1 必测接口
- 用户故事新增
- 用户故事查询列表
- 用户故事编辑
- 用户故事删除
- 创建迭代
- 查询迭代列表
- 将用户故事加入迭代
- 更新故事状态
- 新增回顾
- 查询燃尽图数据

## 12.2 测试覆盖率说明

报告中可按以下方式计算：

`核心接口覆盖率 = 已覆盖核心接口数 / 总核心接口数 × 100%`

### 示例
若定义核心接口共 10 个，测试覆盖 9 个，则覆盖率为 90%。

## 12.3 测试脚本建议

若时间紧，优先选择 Postman + Newman：
- 上手快
- 容易截图
- 适合作业展示

## 15. 报告写作建议

报告重点不是“我用了 AI”，而是“我怎么控制 AI”。

建议重点写：
- 为什么选 Builder 模式
- 为什么选该技术栈
- AI 在哪些环节提高了效率
- AI 在哪些环节产生了错误
- 人做了哪些关键决策
- 测试发现了哪些问题
- 如果再做一次，会怎样优化开发流程

## 16. 本地使用 Codex 与 Trae IDE 的建议方式

## 16.1 Codex 建议用途

适合让 Codex 做：
- 阅读项目结构并指出问题
- 检查接口命名是否一致
- 检查测试是否覆盖核心接口
- 检查前后端字段是否匹配
- 检查 README、部署说明、测试说明是否完整
- 帮你发现潜在 bug

### 使用建议
在项目根目录提供以下文档给 Codex：
- 本文件 `项目执行与技术说明.md`
- `README.md`
- 后端接口文档
- 数据库 schema 文件

如果要进一步提升 Codex 的检查质量，建议另建 `AGENTS.md`，写清楚：
- 项目目标
- 技术栈
- 运行命令
- 测试命令
- 不允许改动的约束
- 你希望它重点检查什么

## 16.2 Trae IDE 建议用途

适合让 Trae IDE 做：
- 根据当前文档生成页面骨架
- 生成基础 CRUD 接口
- 生成表单和列表页面
- 生成测试脚本初稿
- 修复小型 bug
- 根据你的提示重构局部代码

### 使用原则
1. 一次只让 AI 做一个小任务。
2. 提示词里必须给上下文。
3. 生成后先看 diff，再决定是否接受。
4. 不要一次让 AI 同时改前端、后端、数据库和测试。
5. 每次改动后都运行项目和测试。

## 20. 结论

本项目的正确策略不是追求功能炫技，而是：

**基本版功能完整 + 过程记录完整 + 测试可运行 + 报告可分析。**

只要你把范围控制住，把开发顺序排好，把 AI 当工具而不是当神谕，这个项目完全可以由个人主导稳稳做完。
