# Trae 证据链补全方案

本文档用于解决当前项目的一个核心风险：仓库已经具备前后端基础版代码框架，但课程要求明确强调 `Trae IDE Builder 模式` 的过程留痕。因此后续不能伪造“全部代码均由 Trae 从零生成”，而应采用真实、可追溯的两阶段写法。

## 1. 当前事实

- 当前仓库已经完成基础版前后端框架、核心接口、前端页面、接口自动化测试、README、AI 错误修正记录和阶段截图。
- 前期工作中，`Codex` 参与了文档阅读、范围收敛、接口一致性检查、初版代码生成、测试补充和材料整理。
- 目前尚未形成足够完整的 `Trae Builder` 会话记录、过程截图、Gitee 远程仓库记录和双人配合记录。

## 2. 推荐解释口径

正式报告和答辩时，建议统一采用以下口径：

1. 本项目整体采用 Builder 思路推进。
2. 前期先以 Codex 为辅助工具完成文档收敛、原型搭建和基础验证，形成“可运行原型基线”。
3. 在满足课程要求的正式实验推进阶段，转入 `Trae IDE Builder` 模式，围绕该基线继续进行增量开发、交互完善、数据库接入、Bug 修复、测试补充和证据链记录。
4. 人始终负责需求拆解、模型选择、AI 输出审核、错误修正和最终确认。

这个口径的优点是：

- 真实，不伪造工具使用历史。
- 能解释为什么当前仓库已经有代码。
- 能把接下来的 Trae 过程自然衔接到现有项目上。
- 仍然符合“应用 Trae IDE Builder 模式完成课程实验”的主要求。

## 3. 绝对不要这样写

- 不要写“所有代码都是 Trae 从零生成的”。
- 不要把 Codex 生成的内容硬说成 Trae 的会话结果。
- 不要为了凑证据链补伪造的 Builder 对话截图。
- 不要把后补的材料写成“早期就已经存在”。

## 4. Trae 阶段必须做成的 5 个实质性任务

为了让后续 Trae 证据链有说服力，建议不要只在 Trae 里写文档，而要完成真正的代码与功能迭代。优先级如下：

### 任务 1：把“故事加入迭代”改成拖拽式

- 当前状态：已支持“选择故事加入迭代”，但不完全符合文档中“拖拽加入当前迭代”的描述。
- 目标：在 `frontend/src/views/sprints/SprintsView.vue` 中实现更贴近课程要求的拖拽式交互。
- 证据价值：这是课程功能要求里的细节增强，适合在 Trae 中做 1 次完整小任务。

### 任务 2：把后端持久化从 JSON 切换到 MySQL

- 当前状态：仓库用 `backend/data/db.json` 作为默认运行持久化，同时保留了 `backend/sql/schema.mysql.sql`。
- 目标：接入真实 MySQL，补环境变量和连接层，使课程中的“数据库”要求更扎实。
- 证据价值：这是最能体现“增量开发”而不是“只修文档”的任务。

### 任务 3：补充 Trae 下的 Bug 修复闭环

- 目标：在 Trae 中完成 1~2 个真实小 bug 修复，例如：
  - sprint 删除后页面联动刷新
  - 回顾保存后的列表同步刷新
  - 燃尽图无数据时的空态提示
- 证据价值：适合形成“指令 -> AI 输出 -> 人工修正 -> 复测”的完整案例。

### 任务 4：补充测试或测试材料

- 目标：在 Trae 中完成以下二选一：
  - 增加更多 Supertest 场景
  - 额外补一套 Postman/Newman 测试材料，方便报告截图
- 证据价值：能增强测试章节，也更贴合课程“自动化测试”要求。

### 任务 5：补充 Gitee 与最终材料

- 目标：建立远程仓库、推送代码、截图仓库首页、提交历史、分支或 release 信息。
- 证据价值：补齐版本管理要求，是老师很容易检查的一项。

## 5. 每个 Trae 任务要保留的证据

每个任务至少保留下列 5 类证据：

1. `Trae Builder 界面截图`
   需要看到 Builder 面板、当前模型、当前仓库。
2. `用户提示词截图`
   要保留你给 Trae 的任务说明。
3. `AI 输出或 diff 截图`
   要能看出 Trae 实际修改了哪些内容。
4. `运行或测试结果截图`
   证明修改后确实生效。
5. `Git 提交截图`
   形成从任务到提交的闭环。

## 6. 推荐的截图命名方式

建议后续在 `Screenshot/` 中按如下命名：

- `10_trae_model_select.png`
- `11_trae_story_to_sprint_prompt.png`
- `12_trae_story_to_sprint_diff.png`
- `13_trae_story_to_sprint_result.png`
- `14_trae_story_to_sprint_commit.png`
- `15_trae_mysql_prompt.png`
- `16_trae_mysql_diff.png`
- `17_trae_mysql_result.png`
- `18_trae_mysql_commit.png`

这样报告里可以直接按编号插图，不容易乱。

## 7. Builder 中的模型建议

- 默认代码任务：`Doubao-Seed-2.0-Code`
- 说明/总结任务：`DeepSeek-V3.1-Terminus` 或 `GLM-5.1`
- `Auto Mode` 建议关闭，避免自动执行过多操作而不利于过程留痕。

## 8. 最终报告里的写法建议

在正式报告中可写成：

> 项目初期先结合 Codex 对课程文档、接口边界和数据模型进行梳理，并搭建最小可运行原型；随后在 Trae IDE Builder 模式中围绕该原型继续完成交互完善、数据库接入、缺陷修复、测试补充和过程留痕。整个过程中由人类主导需求拆解、AI 输出审核与错误修正，AI 作为虚拟协作成员参与开发，而不是替代人的决策。

## 9. 结论

当前项目最合理的证据链策略不是“重来一遍”，而是：

`承认已有原型 -> 说明其来源 -> 在 Trae 中完成实质性增量开发 -> 保留完整 Builder 证据链 -> 在报告中如实解释工具协同关系。`
