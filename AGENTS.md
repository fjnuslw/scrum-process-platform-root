# AGENTS.md

本文件用于为 Codex 提供项目级约束。Codex 在开始工作前应先阅读本文件，再阅读 `README.md` 与 `docs/` 下的文档。

## 1. 项目背景

这是一个课程实验项目，目标是开发 Web 版 Scrum 过程管理平台。重点不只是功能实现，还包括：

- AI 辅助开发过程留痕
- Git 版本管理
- 接口自动化测试
- AI 生成错误的人工修正记录
- 实验总结材料整理

## 2. 当前阶段目标

默认优先级如下：

1. 保证基本版功能完整
2. 保证代码可运行
3. 保证接口设计清晰
4. 保证测试可执行
5. 保证实现简单、易解释、易写报告

除非明确要求，否则不要主动扩展增强版功能。

## 3. 技术约束

- 前端默认：Vue 3 + Vite + Element Plus
- 后端默认：Node.js + Express
- 数据库默认：MySQL
- 接口风格：RESTful
- 图表默认：ECharts
- 代码风格：清晰、稳定、以课程作业可维护性为优先

如果仓库实际技术栈与上述不同，应以仓库现有实现为准，不要强行重构。

## 4. 工作方式要求

Codex 在执行任何任务时必须遵守：

- 先读项目结构，再提出改动
- 优先小步修改，避免一次性大规模重写
- 先分析影响范围，再改代码
- 改动后说明修改了哪些文件、解决了什么问题、还有哪些风险
- 如果需要新增依赖，先说明原因
- 如果需求不明确，优先采用最小可行实现

## 5. 功能范围边界

必须优先支持的模块：

1. 用户故事管理
2. 产品待办列表管理
3. 迭代计划管理
4. 故事看板管理
5. 燃尽图
6. 迭代回顾管理

非必要不要添加：

- 权限系统
- 复杂登录注册
- 消息通知
- 大而全的统计看板
- 复杂缓存或微服务设计
- 任何课程要求之外的大型增强功能

## 6. 数据与接口要求

- 数据字段命名尽量统一
- 枚举值集中管理
- 拖拽排序必须落库或持久化
- 看板状态变更必须能驱动燃尽图数据更新
- 删除操作要考虑关联数据影响
- 接口响应格式保持统一

## 7. 测试要求

优先为以下接口补测试：

- 用户故事 CRUD
- 待办列表查询与排序更新
- 迭代 CRUD
- 用户故事加入迭代
- 看板状态更新
- 回顾记录新增与查询

如果测试失败，需要优先定位：

1. 是接口实现问题
2. 是字段命名不一致
3. 是测试数据准备问题
4. 是前后端约定不一致

## 8. 输出要求

每次回答尽量包含：

- 对当前问题的理解
- 计划修改哪些文件
- 修改后如何验证
- 潜在风险或待办

如果完成了代码修改，最后给出：

- 已修改文件列表
- 建议执行的命令
- 建议下一步任务

## 9. 禁止事项

- 不要自行删除大量文件
- 不要未经说明就引入复杂架构
- 不要把项目重构成与课程目标无关的产品
- 不要为了“更先进”而增加不必要复杂度
- 不要捏造测试结果、运行结果或已完成状态
